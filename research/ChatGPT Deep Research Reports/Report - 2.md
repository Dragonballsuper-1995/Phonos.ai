# AI-Powered Smartphone Recommender System – Technical Feasibility Study

## System Architecture & Tech Stack  
Modern AI systems rely on modular, scalable stacks. For the *frontend*, a React (or Next.js) web app with TypeScript is recommended (or a cross-platform UI if targeting mobile). The *backend* can be a microservices setup in Python (e.g. FastAPI or Django) or Node.js (Express/NestJS). One service handles query routing and LLM orchestration; another handles data retrieval. Containerisation (Docker/Kubernetes) and deployment on cloud (AWS/GCP) ensure production readiness.  

The core **LLM and RAG components** form the “brain.” We suggest a large pretrained LLM (e.g. OpenAI GPT-4 via API or a high-capacity open-source model like Llama 3/Mistral) for generation, paired with a vector database to store embeddings of our data. Common RAG frameworks (LangChain, LlamaIndex) can orchestrate this. An *embedding model* (such as OpenAI’s text-embedding-3 or a similar high-quality model) encodes text into vectors. The **vector DB** then indexes these embeddings; options include Pinecone (fully managed, scales to billions of vectors), Weaviate or Milvus (open-source, support hybrid search), Qdrant (Rust-based, efficient filtering), or pgvector (Postgres extension). For example, Pinecone offers ease of use for zero-ops “semantic search”, whereas Weaviate gives built-in keyword+vector search. We might choose Pinecone (free tier) to focus on features, or Qdrant for cost-effective self-hosting. 

The **data storage** layer includes:  
- A relational or NoSQL DB (e.g. PostgreSQL) for structured data (phone specs, prices, user profiles).  
- A vector DB for embeddings and unstructured text (scraped reviews, transcripts, spec chunks).  
- Possibly object storage (S3/GCS) for raw scraped files or backups.  

The **integration layer** glues everything together: when a user query arrives, the backend classifies it (Easy/Medium/Deep) and invokes the appropriate logic. For retrieval, the system uses the embedding model + vector DB to fetch relevant chunks of data. These chunks are then inserted into an LLM prompt. As IBM’s guide notes, the *integration layer* “takes the retrieved results, organizes them and inserts them into the model’s input”. Prompt templates can be implemented via LangChain or custom code.

Key tech stack summary (illustrative):  
- **Frontend:** React or Next.js (TypeScript), UI libraries (e.g. Material-UI).  
- **Backend:** Python (FastAPI/Django) or Node (Express/Nest) with async workers. Use container orchestration.  
- **AI Integration:** LangChain or similar to manage LLM calls, embeddings, prompt templates. Use Hugging Face Transformers or OpenAI API for LLM; embed models like `sentence-transformers` or OpenAI embeddings.  
- **Vector DB:** Pinecone (SaaS) or Weaviate/Milvus/Qdrant (self-host). These store embeddings and support ANN search with metadata filtering.  
- **Data Storage:** Postgres/NoSQL for structured specs and user data; S3 for raw files.  
- **Other:** Docker, Kubernetes/ECS/EKS for deployment; Git/GitHub for version control; CI/CD pipelines; monitoring/logging (Prometheus, Grafana, etc).  

This architecture follows best practices: an LLM “generator” consumes context assembled by a retriever and prompt layer. The system scales by sharding the vector store and horizontally scaling query workers. Using a managed LLM API (e.g. GPT-4) accelerates development, but an open LLM allows more control (no vendor lock-in). 

## The Data Ingestion Pipeline  
We build ETL pipelines for each source: **GSMArena**, **YouTube transcripts**, and **official brand sites** (or aggregated APIs). 

- **GSMArena (Specs/Benchmarks):** Use Python (BeautifulSoup or Requests-HTML) to crawl GSMArena. First, gather all brand and model URLs (as shown in a community script). Then iterate each phone page to extract specs (battery capacity, camera MP, weight, CPU, etc) and benchmark scores. The referenced script even uses a proxy IP list to avoid blocks. Key practices: respect robots.txt, throttle requests, and rotate proxies/user-agents to bypass anti-scraping measures. Store cleaned specs in a database (normalise units, e.g. “6.1 inch”→6.1, “4000 mAh”→4000). Merge or update records for new releases.  
  - *Bottlenecks:* GSMArena may block rapid requests or change HTML layout. Mitigation: slow crawl (sleep between requests), use rotating proxies (the `ip_generator.py` example collects proxy IPs to cycle through), and robust parsers (try/except with log of failed pages). If scrapers fail, consider using MobileAPI.dev or a similar device specs API as a fallback. MobileAPI.dev, for example, returns structured JSON for 31K+ devices (specs, prices, images) with one REST call.

- **YouTube Tech Review Transcripts:** First identify relevant videos (e.g. via YouTube Data API search for “smartphone review”). Then fetch transcripts. A popular Python library, **`youtube-transcript-api`**, retrieves a video’s subtitle (including auto-generated) via its ID without a headless browser. Alternatively, using a dedicated service (e.g. Scrapingdog’s YouTube Transcript API) returns a clean JSON transcript in one call, avoiding fragile HTML scraping. Store each transcript text with metadata (video title, channel, date). Clean transcripts by removing timestamps and filler (“um”, repeated phrases). Segment long transcripts into chunks (e.g. 500–1000 tokens) for embedding. Perform sentiment analysis or keyword extraction on segments to capture battery/camera opinions.  
  - *Bottlenecks:* Not all tech reviews have accurate captions; some have no transcripts. For missing ones, fallback to ASR (speech-to-text) might be needed. Also, YouTube API has daily quotas – mitigate by caching transcripts and staggering requests. An unofficial transcript API may bypass quota. Use exponential backoff on failures and keep an audit of which videos were processed.  

- **Official Brand Sites / APIs:** For MSRP and official claims, one could scrape manufacturer sites (Apple, Samsung, etc.) but these are often dynamic. A more maintainable approach is to use a unified device specs/pricing API. MobileAPI.dev (free tier) provides normalized specs and current prices for tens of thousands of phones. Using such an API sidesteps HTML issues. If scraping is still needed, employ headless browsers (Selenium/Playwright) to render pages and extract fields. Standardize pricing by currency.  
  - *Bottlenecks:* Brand sites change layouts, and heavy JS can break scrapers. Prefer official data feeds or APIs. For example, product launch press releases or tech press APIs may also be used.

After collection, **cleaning & storage**: All raw data should be cleaned and normalized (e.g. units, naming conventions). Ingested text (spec pages, transcripts) is split into “chunks” (~512 tokens) and embedded. Each chunk, along with metadata (phone model, source URL, channel name), is stored in the vector DB. This ensures the RAG retriever can fetch relevant facts. Numeric specs go into structured tables (e.g. SQL) for fast filtering. Maintaining consistency (for example, mapping “iPhone 11” vs “Apple iPhone 11”) is critical.

## The Routing Logic  
We design a **tiered routing system** to interpret user queries and tailor the response strategy. First, we classify the query into Tier 1 (Persona-based), Tier 2 (Specs-based), or Tier 3 (Psychographic/Historical). This can be done with simple heuristics (e.g. presence of budget/persona keywords vs. detailed specs vs. personal context) or with a small intent-classifier LLM. 

- **Tier 1 (Easy – Persona-driven):** These queries describe a user persona and budget (e.g. “I’m a gig worker on a $400 budget”). The prompt template for LLM could be: *“You are a phone advisor. The user is [persona description] with a budget of $X. Based on that, recommend a phone and explain why.”* Here, the LLM mainly uses general knowledge. We can still augment with retrieval: e.g. fetch a few spec summaries of known good budget phones or synthesize prior knowledge. The prompt emphasizes persona needs (“good battery for long hours,” “rugged build,” etc.). If desired, a simple rule-based filter on the specs DB (e.g. select phones ≤ budget sorted by battery capacity) can seed the answer. The LLM then rephrases with context.  

- **Tier 2 (Medium – Spec-driven):** Queries specify features and budget (e.g. “best battery and camera under $800”). The system should parse out the attributes and use them to retrieve relevant data. One approach: parse the query into filters (e.g. battery≥X, camera≥Y, price≤800). Apply these filters directly on the specs database to get candidate phones. Alternatively, turn the query into an embedding and retrieve top-matching spec-document chunks from the vector DB. After retrieving data (e.g. spec snippets of candidates), the LLM is prompted: *“User wants strong battery and camera under $800. Here are data on phones matching those criteria: [insert retrieved spec facts]. Recommend the top phone(s) from these and explain.”* In essence, Tier 2 leans on factual lookup: the LLM’s job is to reason over the filtered results and articulate a recommendation.  

- **Tier 3 (Deep – Psychographic/Historical):** These queries include personal history and preferences (e.g. “Upgrading from iPhone 11, hate heavy phones, love gaming”). We treat this as a personalized RAG query. The logic: first incorporate the user’s background into the prompt context. For example: *“The user currently has an iPhone 11 (weight ~194g) and dislikes heavy phones. They are passionate about mobile gaming. Upgrading from their phone, which new smartphone do you recommend and why?”* Then retrieve relevant facts: e.g. spec data of iPhone 11 (to set a baseline) and data on current gaming phones (high GPU, battery). We might query the vector store with terms like “iPhone 11 weight” and “best gaming smartphone 2026” to bring in external info. The LLM will use these plus the embedded context to answer. Essentially, Tier 3 prompts ask the LLM to **think step-by-step**: acknowledge the old phone specs, filter out heavy devices, prioritize gaming performance, and then recommend a model. Prompt engineering here is key: the system might use multi-turn output (e.g., first ask the LLM to list constraints, then to match phones). The integration layer ensures retrieved evidence (from transcripts like “this phone is heavy”) is in the prompt. As IBM notes, the retriever embeddings and prompts “form a single sequence” for the model.

In practice, the backend might first check for persona keywords or personal context to route the query. Each tier uses a tailored prompt template. For example, Tier 1 might not need extensive retrieval (it relies on the persona prompt and general knowledge), while Tier 2/3 use RAG: retrieving phone spec chunks and customer review snippets to ground the answer. Throughout, care is taken to format prompts cleanly (avoiding hallucinations) and to keep the LLM’s instructions clear. 

## Critical Challenges & Mitigation  

- **LLM Hallucinations / Accuracy:** LLMs often “hallucinate” facts if not grounded. Mitigation is core to our design: **RAG**. By retrieving real data and injecting it into the prompt, we force the model to base answers on facts. As NVIDIA explains, RAG “reduces hallucinations by providing the LLM with relevant factual information”. For example, if recommending a phone, we only include its actual specs from our DB. We can also instruct the LLM to only answer from given data and to admit when uncertain. Additional solutions: fine-tune or prompt the model on tech domain data, use answer-verification (e.g. have a second model check facts), and limit answers to what retrieval returned.  

- **Web Scraping & Data Collection Blocks:** Scraping GSMArena or video sites can trigger anti-bot measures. In our GSMArena pipeline, we plan rotating proxies and IP pools as shown in existing scraper code. Similarly, YouTube scraping should favor APIs over raw HTML: a transcript API (Scrapingdog or `youtube-transcript-api`) yields structured JSON and sidesteps HTML parsing. Always respect robots.txt and throttle request rates. If scraping fails, we use paid APIs or data services. Caching results locally prevents repeated hits to the same page. In short, use proxies and official endpoints whenever possible to avoid blocks.

- **Rate Limits & Latency:** External services (YouTube API, LLM API, vector DB) impose limits. The solution is caching and batching: store transcripts/specs so we rarely query the same data twice. Break ingestion into asynchronous jobs and use background workers. For real-time queries, optimize vector search (use HNSW indexes as recommended) and limit prompt context size. Note that RAG adds latency (embedding + search + prompting); to mitigate, we pre-compute embeddings offline and use a fast ANN search. If using an LLM API, handle rate limits with exponential backoff and queuing. Finally, ensure the system can scale horizontally: multiple workers can process requests in parallel. These steps balance responsiveness with external constraints. 

*(Other considerations:* data cleaning heterogeneity, context window limits, and ongoing maintenance are also challenges. For example, we must keep the spec database up-to-date with new phone releases. Regular pipeline jobs and modular design will address these.*)
